# upstairs — 要件定義書

## 1. 概要

Windows エクスプローラーのファイル一覧の「余白」(アイテムのない領域) をダブルクリックすると、
1 つ上の階層フォルダーへ移動できるようにする常駐型ユーティリティ。

昔のファイラーや macOS 系ツールにある「余白ダブルクリックで上へ」の操作感を
Windows 11 の標準エクスプローラーで再現することが目的。

## 2. 対象環境

| 項目 | 内容 |
|---|---|
| OS | Windows 11 (開発機: 10.0.26200)。可能なら Windows 10 も動作対象 |
| 対象アプリ | 標準エクスプローラー (`CabinetWClass`) のみ |
| 権限 | 管理者権限不要で動作すること |

## 3. 機能要件

### 3.1 基本動作 (必須)

- **F-1**: エクスプローラーのファイル一覧ペインの余白を**左ダブルクリック**すると、上の階層へ移動する。
  - 移動は `Alt + ↑`(上へ) と同じ挙動とする (エクスプローラー標準のナビゲーション履歴に乗る)。
- **F-2**: ファイル・フォルダーなどの**アイテム上でのダブルクリックには反応しない** (通常の「開く」動作を妨げない)。
- **F-3**: 複数のエクスプローラーウィンドウが開いていても、**クリックされたウィンドウだけ**が移動する。
- **F-4**: ドライブルート (例 `C:\`) など、これ以上上がれない場所では何もしない (エラーも出さない)。

### 3.2 反応してはいけない領域 (必須)

余白判定はファイル一覧ペインの背景に限定し、以下では発動しない:

- **F-5**: デスクトップ (デスクトップもエクスプローラーだが対象外)
- **F-6**: ナビゲーションペイン (左のツリー)、アドレスバー、リボン/コマンドバー、検索ボックス、ステータスバー
- **F-7**: 列ヘッダー (詳細表示のとき)、グループ見出し
- **F-8**: 他アプリの「ファイルを開く/保存」ダイアログ (v1 では対象外。将来オプション化を検討)
- **F-9**: 名前変更のインライン編集中のテキストボックス

### 3.3 誤発動の防止 (必須)

- **F-10**: 余白からの**ドラッグ操作** (ラバーバンド範囲選択) は妨げない。ダブルクリック判定は OS のダブルクリック時間・許容移動距離 (`GetDoubleClickTime` / `SM_CXDOUBLECLK`) に従う。
- **F-11**: `Ctrl` / `Shift` などの修飾キーを押しながらのダブルクリックでは発動しない (選択操作と競合するため)。

### 3.4 常駐・操作 (必須)

- **F-12**: タスクトレイに常駐する。トレイアイコンのメニューから以下を操作できる:
  - 有効 / 無効の切り替え (一時停止)
  - Windows 起動時に自動開始 (ON/OFF)
  - 終了
- **F-13**: 多重起動を防止する (二重に起動した場合は既存プロセスを優先して終了)。

### 3.5 オプション (v2 以降の候補)

- 余白の**ミドルクリック**や**トリプルクリック**への割り当て変更
- 「上へ」の代わりに「戻る」を割り当てる設定
- ファイルダイアログ (F-8) への対応

## 4. 非機能要件

- **N-1 軽量性**: 常駐時の CPU 使用率はほぼ 0%、メモリは目安 50MB 以下。マウスフックのコールバックは即時リターンし、重い判定 (UI Automation 等) はフック外のスレッドで行うか、判定を最小化する。フック遅延によるマウス全体のもたつきを発生させない。
- **N-2 安定性**: エクスプローラーの再起動 (explorer.exe 再起動) 後もそのまま動作し続けること。
- **N-3 安全性**: ファイル操作・シェル拡張の注入 (DLL injection) は行わない。エクスプローラーをクラッシュさせるリスクのある手法は採らない。
- **N-4 配布**: 単一 exe (自己完結) で配布できること。インストーラー不要。

## 5. 技術方式 (確定)

| 要素 | 方式 |
|---|---|
| 言語/ランタイム | **C# / .NET 8** (WinForms ベースのトレイ常駐アプリ、self-contained 単一 exe 発行) ※確定 |
| クリック検出 | 低レベルマウスフック `WH_MOUSE_LL` でダブルクリックを検出 |
| ウィンドウ判定 | `WindowFromPoint` → トップレベルのクラス名が `CabinetWClass` であること |
| 余白判定 | **UI Automation** でカーソル位置の要素を取得し、要素がリストアイテム (`ListItem`/`TreeItem` 等) でなく、ファイル一覧 (`UIItemsView` / `DirectUIHWND` / `SysListView32` 配下) の背景であることを確認 |
| 階層移動 | 対象ウィンドウへ `Alt + ↑` を送出 (`SendInput`)。代替: `Shell.Application` COM でウィンドウを HWND 照合して `Navigate` |

### 判定フロー

```
ダブルクリック検出 (WH_MOUSE_LL)
  → カーソル下のトップレベルウィンドウが CabinetWClass か?  … No → 何もしない
  → 修飾キーが押されていないか?                              … No → 何もしない
  → UI Automation でカーソル下の要素を取得
       アイテム/ヘッダー/編集ボックス等か?                    … Yes → 何もしない
       ファイル一覧の背景 (余白) か?                          … Yes → Alt+↑ を送出
```

### 既知のリスク

- Windows 11 のファイル一覧は `DirectUIHWND` / WinUI ベースで、Windows Update により
  UI Automation ツリー構造が変わる可能性がある → クラス名・ControlType の判定は複数パターンを許容する。
- タブ機能 (Win11) 使用時も、クリック座標のあるアクティブタブに対して Alt+↑ が効くことを確認する。

## 6. 受け入れ条件 (テスト観点)

1. 詳細/一覧/大アイコンの各表示モードで、余白ダブルクリック → 親フォルダーへ移動する
2. ファイル/フォルダーのダブルクリック → 通常どおり開く (二重発動しない)
3. デスクトップの余白ダブルクリック → 何も起きない
4. ナビゲーションペイン・アドレスバー・列ヘッダーのダブルクリック → 移動しない
5. `C:\` 直下で余白ダブルクリック → 何も起きない
6. 余白からのドラッグ範囲選択 → 正常に動作する
7. 2 枚のエクスプローラーを並べ、片方だけ操作 → 操作した側だけ移動する
8. トレイメニューで「無効」→ 発動しなくなる。「有効」で復帰する
9. explorer.exe を再起動 → 引き続き動作する
10. 動画再生などマウス多用時にカーソルのもたつきがない
